# 감상 기록과 리뷰 작성 플로우 통합

## 목표

상세 페이지에서 `감상한 작품` 선택과 `리뷰 작성` 버튼을 같은 감상 기록 플로우로 통합한다.

- `감상할 작품`은 기존처럼 드롭다운에서 즉시 저장한다.
- `감상한 작품`은 `감상 날짜 + 별점 필수 + 코멘트 선택`을 받는 모달로 기록한다.
- 기존에 이미 `watched`지만 리뷰가 없는 항목은 현재처럼 리뷰 작성 진입을 유지하되, 모달의 날짜 기본값으로 기존 `watchedAt`을 사용한다.
- 리뷰 수정에서도 같은 `watchedAt` 값을 함께 다룬다.
- 리뷰 좋아요 초기화는 `rating` 또는 `comment`가 바뀔 때만 유지하고, 날짜만 바뀌는 경우에는 초기화하지 않는다.
- `watchlist`와 `reviews` 테이블은 합치지 않고, UI와 저장 흐름만 통합한다.

## 구현 반영 후 구조 요약

- `ReviewFormModal`은 신규 작성과 수정 모두에서 `watchedAt`, `rating`, `comment`를 함께 저장한다.
- `ReviewsService.create()`는 리뷰 생성 트랜잭션 안에서 워치리스트를 `watched`로 자동 등록하고, 이미 `watched` 기록이 있으면 전달된 `watchedAt`으로 날짜를 유지/갱신한다.
- `ReviewsService.update()`는 `watchedAt`을 받으면 같은 트랜잭션에서 해당 유저/콘텐츠의 워치리스트를 `watched`로 유지하고 날짜를 갱신한다.
- `WatchlistStatusButton`은 상세 페이지에서 `감상할 작품`은 즉시 저장하고, `감상한 작품`은 공유 `ReviewFormModal`을 연다.
- 신규 감상 기록의 날짜 기본값은 UTC가 아니라 한국 날짜(`Asia/Seoul`) 기준 오늘로 계산한다.
- `WatchlistStatusButton`은 `감상한 작품` 제거 시 기존 리뷰를 확인하고, 리뷰가 있으면 리뷰도 함께 삭제된다는 확인 모달을 거친 뒤 워치리스트를 삭제한다.
- 백엔드는 워치리스트 제거 시 해당 유저/콘텐츠 리뷰도 같은 트랜잭션에서 삭제하고, 리뷰가 있는 작품의 `want_to_watch` 등록을 차단한다.
- `ReviewFormWrapper`는 `watchlist-updated` 이벤트를 수신하면 내 리뷰를 다시 조회해, 상세 페이지 드롭다운에서 감상 기록을 저장한 직후 리뷰 카드가 바로 보이게 한다.
- `WatchlistCard`는 내 기록의 `watched` 카드에서 리뷰가 없으면 `리뷰 작성` 버튼을 유지하고, 기존 `watchedAt`을 날짜 기본값으로 전달한다.
- `WatchlistCard`의 날짜 영역은 기존 위치와 형태를 유지하되, 단독 날짜 모달 대신 공유 `ReviewFormModal`을 연다.
- `WatchlistCard`와 `ReviewForm`은 좋아요 버튼의 최신 로컬 카운트를 수정 모달에 넘겨, 사용자가 방금 누른 좋아요도 초기화 경고 조건에 반영한다.
- 리뷰 좋아요 초기화는 `rating` 또는 `comment`가 실제로 바뀔 때만 수행하고, `watchedAt`만 변경하는 경우에는 초기화와 경고 문구를 모두 생략한다.
- 기존 `WatchedDateModal` 파일은 남아 있지만 현재 상세/내 기록의 감상 기록 수정 플로우에서는 사용하지 않는다.

## Phase 1: 상세 페이지 watched 진입을 리뷰 모달로 통합

변경 파일:
- `frontend/src/components/watchlist/WatchlistStatusButton.tsx`
- `frontend/src/components/review/ReviewFormModal.tsx`
- `frontend/src/app/contents/[type]/[tmdbId]/page.tsx`
- 필요 시 `frontend/src/components/review/ReviewFormWrapper.tsx`

작업:
- `WatchlistStatusButton`이 `contentId`를 받을 수 있게 상세 페이지 전달 경로를 확장한다.
- `감상한 작품` 선택 시 `WatchedDateModal` 대신 `ReviewFormModal`을 연다.
- `want_to_watch` 저장과 제거 흐름은 기존 API를 유지한다.
- 신규 감상 기록 모달에서는 날짜 입력, 별점, 코멘트를 한 번에 입력한다.
- 모달 저장 후 `watchlist-updated` 이벤트와 `router.refresh()`로 상세 페이지 버튼 상태와 리뷰 영역을 갱신한다.
- 버튼/모달 문구는 신규 작성 시 `감상 기록` 또는 `기록하기`처럼 리뷰와 감상 기록을 모두 포괄하도록 정리한다.

검증:
- 상세 페이지에서 미등록 상태의 `감상한 작품` 선택 시 감상 기록 모달이 열리는지 확인한다.
- 저장 후 워치리스트 상태가 `watched`로 바뀌고 내 리뷰가 생성되는지 확인한다.
- `감상할 작품` 선택은 기존처럼 즉시 저장되는지 확인한다.

## Phase 2: 기존 watched 항목의 리뷰 작성 날짜 기본값 처리

변경 파일:
- `frontend/src/components/review/ReviewFormModal.tsx`
- `frontend/src/components/watchlist/WatchlistCard.tsx`
- `frontend/src/types/watchlist.ts`
- 필요 시 `frontend/src/types/review.ts`

작업:
- `ReviewFormModal`에 `initialWatchedAt` 같은 선택 props를 추가한다.
- 내 기록의 `WatchlistCard`에서 리뷰가 없는 `watched` 항목의 `리뷰 작성` 버튼은 현재처럼 유지한다.
- `WatchlistCard`가 `ReviewFormModal`을 열 때 해당 항목의 `watchedAt`을 전달한다.
- 내 기록 카드의 기존 날짜 영역 버튼은 유지하되, 단독 날짜 모달 대신 공유 `ReviewFormModal`을 여는 감상 기록 수정 진입점으로 바꾼다.
- 리뷰가 없는 `watched` 카드에서 날짜 영역을 누르면 리뷰 작성과 같은 모달을 열고, 리뷰가 있는 카드에서 날짜 영역을 누르면 기존 리뷰 수정 모달을 연다.
- `ReviewFormModal`은 신규 리뷰 작성 시 `initialWatchedAt`이 있으면 날짜 기본값으로 사용한다.
- `initialWatchedAt`이 없고 아직 `watched` 상태가 아니면 기존처럼 오늘 날짜를 기본값으로 사용한다.
- 이미 `watched`인 항목에서 리뷰를 새로 작성할 때도 `watchedAt`을 함께 POST하여 기존 날짜가 보존되도록 한다.

검증:
- 내 기록의 `watched` 카드 중 리뷰가 없는 항목에서 `리뷰 작성`을 누르면 기존 감상일이 모달에 채워지는지 확인한다.
- 내 기록의 기존 날짜 영역 버튼을 누르면 같은 감상 기록 모달이 열리는지 확인한다.
- 저장 후 리뷰가 생성되고 기존 감상일이 바뀌지 않는지 확인한다.
- 기존 `watchedAt`이 없는 예외 데이터는 오늘 날짜 기본값으로 저장 가능한지 확인한다.

## Phase 3: 리뷰 수정에서 watchedAt 함께 수정

변경 파일:
- `backend/src/reviews/dto/update-review.dto.ts`
- `backend/src/reviews/reviews.service.ts`
- `backend/src/reviews/reviews.service.spec.ts`
- `frontend/src/components/review/ReviewFormModal.tsx`
- `frontend/src/components/review/ReviewForm.tsx`
- `frontend/src/components/watchlist/WatchlistCard.tsx`
- 필요 시 `frontend/src/types/review.ts`

작업:
- `UpdateReviewDto`에 선택 필드 `watchedAt`을 추가한다.
- `ReviewsService.update()`가 `watchedAt`을 받으면 같은 트랜잭션에서 해당 유저/콘텐츠의 워치리스트 항목을 `watched`로 유지하고 날짜를 갱신한다.
- 리뷰 수정 모달도 기존 `watchedAt`을 기본값으로 표시한다.
- 상세 페이지의 내 리뷰 수정과 내 기록 카드의 리뷰 수정 진입 모두 같은 날짜 값을 사용한다.
- 리뷰의 `rating` 또는 `comment`가 바뀌는 경우 기존처럼 좋아요 초기화 정책을 유지한다.
- 날짜만 바뀌는 경우 리뷰 좋아요는 초기화하지 않는다.
- 좋아요 초기화 경고 문구는 `rating` 또는 `comment`가 실제로 바뀐 경우에만 표시하고, `watchedAt`만 바뀐 경우에는 표시하지 않는다.

검증:
- 리뷰 수정에서 별점/코멘트만 바꾸면 기존 좋아요 초기화 정책이 유지되는지 확인한다.
- 날짜만 바꾸면 워치리스트 `watchedAt`만 바뀌고 좋아요가 유지되는지 확인한다.
- 날짜만 바꾸는 수정 화면에서는 좋아요 초기화 경고 문구가 표시되지 않는지 확인한다.
- 날짜와 리뷰 내용을 함께 바꾸면 리뷰 수정과 날짜 수정이 모두 반영되는지 확인한다.

## Phase 4: 제거와 수정 UX 정리

변경 파일:
- `frontend/src/components/watchlist/WatchlistStatusButton.tsx`
- `frontend/src/components/watchlist/WatchlistCard.tsx`
- 필요 시 `backend/src/watchlist/watchlist.service.ts`
- 필요 시 `backend/src/watchlist/watchlist.service.spec.ts`

작업:
- 내 기록 카드의 날짜 영역 UI는 기존 위치와 형태를 유지한다.
- 날짜 영역 클릭은 단독 날짜 수정이 아니라 공유 `ReviewFormModal` 기반의 감상 기록 수정으로 통일한다.
- 기존 `WatchedDateModal`은 상세/내 기록의 감상 기록 수정 플로우에서는 제거하고, 다른 사용처가 없으면 후속 정리 대상에 둔다.
- 상세 페이지에서 `watched` 상태를 제거할 때 리뷰가 있으면 리뷰도 함께 삭제된다는 확인 모달을 먼저 표시한다.
- 확인 후에는 워치리스트와 리뷰를 함께 제거한다. 리뷰 삭제는 댓글/좋아요까지 DB cascade로 함께 제거된다.
- 리뷰만 삭제하는 경우에는 감상 기록을 남긴다.
- 리뷰가 남아 있는 작품을 `want_to_watch`로 등록하는 것은 백엔드 검증으로 차단한다.

검증:
- `watched` 제거 후 상세 페이지 버튼 상태가 미등록으로 돌아오는지 확인한다.
- 리뷰가 있는 상태에서 워치리스트 제거 시 확인 모달이 표시되고, 확인 후 리뷰 카드도 사라지는지 확인한다.
- 리뷰가 남아 있는 비정상 상태에서 `감상할 작품` 등록 API가 거절되는지 확인한다.
- 내 기록 카드 날짜 영역에서 공유 모달이 열리고, 날짜/별점/코멘트를 같은 화면에서 수정할 수 있는지 확인한다.

## Phase 5: 구현 후 보완 반영

변경 파일:
- `frontend/src/components/review/ReviewFormWrapper.tsx`
- `frontend/src/components/watchlist/WatchlistStatusButton.tsx`
- `frontend/src/components/watchlist/WatchlistCard.tsx`
- `frontend/src/components/review/__tests__/ReviewFormWrapper.test.tsx`
- `frontend/src/components/watchlist/__tests__/WatchlistStatusButton.test.tsx`
- `frontend/src/components/watchlist/__tests__/WatchlistCard.test.tsx`

작업:
- 상세 페이지 드롭다운의 `감상한 작품` 선택으로 리뷰를 작성한 직후, 리프레시 없이 내 리뷰 카드가 보이도록 `ReviewFormWrapper`가 `watchlist-updated` 이벤트에서 내 리뷰를 재조회한다.
- 상세 페이지에서 워치리스트 제거 시 리뷰가 남지 않도록, 리뷰가 있는 `watched` 항목은 확인 모달을 거쳐 워치리스트와 리뷰를 함께 삭제한다.
- `감상할 작품` 등록은 프론트에서 별도 리뷰 삭제 확인을 거치지 않고 즉시 저장을 시도한다.
- 백엔드는 리뷰가 있는 작품을 `want_to_watch`로 등록하는 요청을 거절하고, 워치리스트 삭제 시 연결된 리뷰를 함께 삭제한다.
- 워치리스트 상태 응답의 `watchedAt`은 감상 기록 모달의 날짜 기본값으로 사용한다.
- 내 기록 카드에서 좋아요를 즉시 다시 누른 뒤 리뷰를 수정하는 경우에도 최신 좋아요 수 기준으로 초기화 경고 문구가 표시되도록 `LikeButton`의 `onChange` 결과를 로컬 상태로 보관한다.
- 리뷰 수정 저장 후에는 로컬 좋아요 수 override를 초기화해 서버 응답의 `likesCount`와 다시 동기화한다.
- 신규 작성의 기본 날짜는 `toISOString()` 대신 `Asia/Seoul` 기준 날짜 유틸을 사용한다.

검증:
- 상세 페이지 드롭다운에서 `감상한 작품` 저장 후 리뷰 카드가 즉시 나타나는지 확인한다.
- 워치리스트 삭제 시 리뷰가 있으면 리뷰 삭제 확인 모달이 먼저 표시되고, 확인 후 리뷰 카드가 사라지는지 확인한다.
- 리뷰가 남아 있는 작품의 `감상할 작품` 등록은 백엔드에서 거절되는지 확인한다.
- UTC 기준 전날로 밀리는 시간대에도 신규 작성 기본 날짜가 한국 날짜 기준 오늘로 표시되는지 확인한다.
- 내 기록에서 좋아요를 누른 직후 리뷰 내용을 수정하면 좋아요 초기화 경고가 표시되고, 날짜만 바꾸면 경고가 표시되지 않는지 확인한다.
- 관련 컴포넌트 테스트를 추가해 이벤트 기반 리뷰 재조회, 리뷰 삭제 확인, 최신 좋아요 수 전달을 고정한다.

## 전체 검증 방법

프론트엔드:
- `cd frontend && npm run lint`
- `cd frontend && npm test`
- `cd frontend && npm run build`

백엔드:
- `cd backend && npm run lint`
- `cd backend && npm test`
- `cd backend && npm run build`

수동 확인:
- 상세 페이지 미등록 상태에서 `감상할 작품` 저장
- 상세 페이지 미등록 상태에서 `감상한 작품` 선택 후 감상 기록 저장
- 상세 페이지 `리뷰 작성` 버튼으로 같은 모달 저장
- 이미 `watched`지만 리뷰가 없는 내 기록 카드에서 기존 `watchedAt` 기본값 확인
- 기존 리뷰 수정 시 `watchedAt` 기본값과 저장 결과 확인

## 리스크

- `WatchlistStatusButton`이 `contentId`를 받는 경로가 추가되었으므로, 상세 페이지 외 다른 사용처에서 `contentId` 누락 시 `watched` 모달 진입을 비활성화하는 방어가 유지되어야 한다.
- 기존 `watched`만 있고 리뷰가 없는 데이터가 남아 있으므로, 모달 기본값과 저장 후 날짜 보존을 계속 회귀 테스트해야 한다.
- 리뷰 수정에서 `watchedAt`까지 함께 처리하므로 백엔드 트랜잭션 범위가 넓어졌고, 날짜 저장 실패가 리뷰 수정과 함께 롤백되는지 유지해야 한다.
- 날짜만 수정하는 경우 좋아요 초기화와 초기화 경고 문구가 발생하지 않도록 변경 조건을 `rating/comment`와 분리한 상태를 유지해야 한다.
- `WatchedDateModal`은 현재 프로덕션 사용처가 사라졌고 테스트만 남아 있으므로, 후속 정리 시 삭제 여부와 테스트 제거를 함께 판단해야 한다.
- `감상한 작품` 제거가 리뷰 삭제까지 포함하므로, 삭제 확인 문구와 백엔드 트랜잭션 동작이 계속 같은 정책을 가리켜야 한다.
- 리뷰가 남아 있는 비정상 상태에서 `want_to_watch` 등록을 막는 백엔드 검증을 유지해야 한다.
